1. `<step name="load_project_state">`: invoke `df-tools verify flutter-ui-bootstrap` at executor start for `type: ui` + `stack: flutter` TRDs. Handle action:warn (emit setup task), action:skip (continue), action:fail (hard-fail).

<anti_patterns>
**Do NOT use `flutter test integration_test/ -d chrome` for web.** Per Pitfall #1: web E2E requires `flutter drive --driver=test_driver/integration_test.dart --target=<tests.integration> -d chrome` (with chromedriver running). This is canonical 2026 docs guidance. The executor MUST use the right invocation per platform.

**Do NOT run Maestro on web.** Per Pitfall #2 + TRD 10-02's "Web verification mechanism" section: Maestro on Flutter web is blocked upstream (mobile-dev-inc/maestro#2591). Maestro is mobile-only BY DESIGN. There is NO `maestro_web: true` opt-in flag — the executor must NOT branch on any such flag. Web verification is entirely via `flutter drive` invoking the same `tests.integration` path that mobile uses via `flutter test`.

**Do NOT fail flutter analyze on pre-existing warnings.** Per Pitfall #6: `flutter analyze` exits non-zero on ANY warning, including pre-existing. Compare against a baseline captured at task start. Only fail the task if NEW warnings appeared.

**Do NOT hard-fail on the FIRST run when bootstrap is missing.** Per REQ-10-07 / TRD 10-04a: graceful first-run via action:warn → emit setup task. The marker is set ONLY AFTER setup completes; the next run will fail hard if infra goes missing.

**Do NOT skip platform iteration when TRD has `platform: [mobile, web]`.** BOTH must run. If chromedriver isn't available, emit a checkpoint asking the user to start it — do NOT silently skip web verification.
</anti_patterns>

<error_recovery>
If `flutter analyze` cannot be invoked because Flutter is not installed on the executor's machine: devflow-claude itself does NOT need Flutter installed. The executor gate only applies when running against a Flutter consumer project. If `command -v flutter` fails AND TRD has `type: ui`: emit a checkpoint asking the user to install Flutter (or run from a project where it IS installed). Do NOT silently skip verification.

If `maestro` is not installed: same as above — emit a checkpoint. Do not silently skip. Maestro install: `curl -fsSL "https://get.maestro.dev" | bash`.

If `chromedriver` is not running when invoking `flutter drive -d chrome`: emit a checkpoint asking the user to start it. Pre-task setup step in the gate: `pgrep chromedriver || echo "chromedriver not running — required for web E2E"`.

If a TRD's `tests.integration` field is missing: the planner gate (TRD 10-03) should have caught this. If it slipped through to the executor, document as a Deferred Issue and escalate per Rule 4.
</error_recovery>

# At task START (capture baseline)

NEW_WARNINGS=$(diff <(echo "$BASELINE_ANALYZE") <(echo "$CURRENT_ANALYZE") | grep '^>')

  echo "FAIL: task introduced new flutter analyze warnings:"

\`\`\`
