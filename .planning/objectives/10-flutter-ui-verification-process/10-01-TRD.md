<anti_patterns>
**Do NOT add new fields to FRONTMATTER_SCHEMAS.trd.required.** Doing so would break every non-Flutter TRD in the repo. The enforcement of "Flutter UI TRDs must have these fields" happens SEMANTICALLY in the planner agent (TRD 10-03, via `PLANNING INCONCLUSIVE`), not via schema validation. This is the explicit guidance from RESEARCH.md "Anti-Patterns to Avoid" → "Forcing Flutter-UI fields onto non-Flutter TRDs."

**Do NOT modify `extractFrontmatter` or any parser internals.** The existing parser already handles nested objects, inline arrays, and block arrays per its line 1-220 logic. Per RESEARCH.md "Don't Hand-Roll" table, the YAML parsing is already in place. Tests in this TRD PROVE that fact; they do not introduce parser changes.

**Do NOT add a new schema entry (e.g., `trd_ui:`).** That would fragment validation. One TRD schema, optional fields. The planner branches semantically.

**Do NOT document `platform: [mobile]` as the default for Flutter UI TRDs.** Both mobile AND web are required coverage. The default for ANY Flutter UI TRD is `platform: [mobile, web]`. A TRD that ships only one platform must explicitly narrow.

**Do NOT introduce a `maestro_web` opt-in flag in schema docs.** Maestro on Flutter web has an upstream blocker (mobile-dev-inc/maestro#2591). The web-tier black-box verifier is `flutter drive` invoking the SAME `tests.integration` path that mobile uses via `flutter test`. `tests.maestro` is mobile-only BY DESIGN, not opt-in.

**Do NOT generate test data with an LLM.** Per `no_llm_test_data` constraint and CLAUDE.md TDD Playbook habit 4: hand-build the fixture YAML strings as inline template literals in the test file. They are short, readable, and deterministic.
</anti_patterns>
